1. PROJECT OVERVIEW
================================================================================

KharchaPay is a production-ready enterprise expense tracking, treasury
management, and procure-to-pay application that combines traditional finance
workflows with Solana blockchain (Token-2022) for verifiable payments and
Circle-powered fiat off-ramping. Built as a monorepo with Next.js 14 (App
Router), PostgreSQL, and enterprise-grade security (CSRF, step-up re-auth,
JWT versioning, AES-256-GCM encryption at rest).

2. AUTHENTICATION FLOW - COMPLETE DETAIL
================================================================================

2.1 LOGIN FLOW
--------------------------------------------------------------------------------
Endpoint: POST /api/auth/login
Body: { email, password }

1. Global rate limit check (checkGlobalLimit - 200/min)
2. Auth-specific rate limit (auth:login key - 10/min)
3. Validate body with loginBodySchema (Zod)
4. Per-email rate limit (isRateLimited) - blocks after 10 failed attempts/60s
5. Find user by email (case-insensitive)
6. Verify password with argon2.verify
7. Create JWT via createToken({ sub: userId, authTime, jwtVersion })
8. Set httpOnly cookie: kharchapay_token (7-day expiry)
9. Return { user: ApiUser }

Errors: 429 (rate limited), 400 (validation), 401 (invalid credentials)


2.2 REGISTER FLOW
--------------------------------------------------------------------------------
Endpoint: POST /api/auth/register
Body: { email, username, password } (password min 8 chars)

1. Validate registerBodySchema
2. Check email + username uniqueness
3. argon2.hash(password)
4. Create User record
5. createToken + setAuthCookie
6. Return { user }


2.3 SESSION / JWT STRUCTURE
--------------------------------------------------------------------------------
Token algorithm: HS256
Payload: { sub: userId, authTime: Unix timestamp, jwtVersion: number }
Expiry: 7 days
Storage: httpOnly cookie kharchapay_token
Secure: true in production; SameSite: lax
Library: jose 5.2.0 (SignJWT / jwtVerify)

JWT Version: User.jwtVersion in DB. On password change, forgot/reset, or
logout-everywhere, jwtVersion is incremented. getCurrentUser() verifies
payload.jwtVersion === user.jwtVersion. Mismatch = invalid session.


2.4 PROTECTED ROUTES
--------------------------------------------------------------------------------
- Layout: src/app/app/layout.tsx calls getCurrentUser()
- Redirect: Not authenticated → /login
- Org status checks:
  - PENDING_PAYMENT → redirect to /onboarding/pay?intentId=...
  - PENDING_TERMS → redirect to /onboarding/terms?orgId=...
- After auth: Renders AppShell (sidebar + layout) wrapped in CsrfAndReauthProvider
  with TreasuryStatusBanner at top
- Middleware: src/middleware.ts - security headers only (no auth logic in middleware)


2.5 ADDITIONAL AUTH ENDPOINTS
--------------------------------------------------------------------------------
POST /api/auth/logout           - Clear auth cookie (requires CSRF)
POST /api/auth/logout-everywhere - Increment jwtVersion (invalidates all sessions)
POST /api/auth/reauth           - Verify password, issue new JWT with fresh authTime
POST /api/auth/change-password  - Verify current, set new, increment jwtVersion
POST /api/auth/forgot-password  - Create PasswordResetToken, dev: log link
POST /api/auth/reset-password   - Validate token, set password, increment jwtVersion
GET  /api/auth/reauth-test      - Test requireRecentAuth (param maxAge)


2.6 STEP-UP RE-AUTH (SENSITIVE ACTIONS)
--------------------------------------------------------------------------------
Purpose: Require recent authentication (within 15 min) for high-risk actions.

REAUTH_MAX_AGE_SECONDS = 15 * 60 (15 minutes)

requireRecentAuth(maxAgeSeconds):
- Reads JWT authTime
- If (now - authTime) > maxAgeSeconds → 403 { code: "REAUTH_REQUIRED" }

Applied to:
- POST .../requests/[id]/pay
- PATCH .../vendors/[id] (wallet, status)
- PUT .../spend-policy
- PUT .../approval-policy
- POST .../reconcile/run, .../reconcile/request
- POST .../demo/reset
- Treasury mutations (payouts, policy, safety controls)

UI: ReauthModal component - if API returns REAUTH_REQUIRED, modal asks
    "Re-enter password", POST /api/auth/reauth { password }, then retries
    original action. Wrapped in CsrfAndReauthProvider context.


2.7 CSRF PROTECTION
--------------------------------------------------------------------------------
Cookie: kharchapay_csrf (2h, httpOnly, SameSite: lax)
Header: x-csrf-token must match cookie on mutations

GET /api/csrf - Sets cookie with 32 random bytes (base64url), returns { csrfToken }
requireCsrf(request) - Validates header against cookie; 403 code: "CSRF" if mismatch
Comparison: Constant-time XOR (safeCompare) to prevent timing attacks

Used on: All POST/PUT/PATCH/DELETE (login/register excluded)
Frontend: fetchWithCsrf() from @/lib/fetch-with-csrf
  - Auto-injects x-csrf-token for mutations
  - On 403 + CSRF code: clears cache, fetches fresh token, retries once

3. USER ROLES & RBAC
================================================================================

3.1 ORG ROLES (OrgRole enum)
--------------------------------------------------------------------------------
| Role     | Write Access | Main Capabilities |
|----------|--------------|-------------------|
| ADMIN    | Yes          | All org settings, members, integrations, QBO, webhooks, payments, vendors, budgets, approval policy, spend policy, audit retention, ops, chain setup, treasury management, payouts, safety controls |
| APPROVER | Yes          | Approve/reject requests, invoice verify/reject, vendor BLOCKED status, onboarding, bank changes (per policy), payout approvals |
| STAFF    | Yes          | Create requests, POs, receipts, invoices; view vendors (incl. ONBOARDING); no approvals |
| AUDITOR  | Read-only    | Dashboard, requests, payments, POs, receipts, invoices, reports, audit log, compliance, docs - NO mutations |


3.2 ROLE ENFORCEMENT
--------------------------------------------------------------------------------
requireOrgRole(orgId, userId, [OrgRole.ADMIN]) - Specific roles
requireOrgReadAccess(orgId, userId) - ADMIN, APPROVER, STAFF, AUDITOR
requireOrgWriteAccess(orgId, userId) - ADMIN, APPROVER, STAFF only; AUDITOR → 403 READ_ONLY_ROLE
requireOrgMember(orgId, userId) - Alias for write-capable members
requireUser() - Throws 401 if not authenticated


3.3 RBAC PERMISSIONS (src/lib/rbac.ts) - 31 PERMISSION KEYS
--------------------------------------------------------------------------------
PERMISSIONS_BY_ROLE maps each role to nav/visibility permissions:

STAFF:     dashboard, chat, new_request, requests, payments, purchase_orders,
           receipts, invoices, reports, setup, docs, vendors

APPROVER:  Same as STAFF + approvals

ADMIN:     All of APPROVER + audit_log, compliance, security_check, ops,
           departments, budgets, members, approval_policy, spend_policy,
           audit_retention, integrations, demo*, solana_ct_demo*, debug*

AUDITOR:   dashboard, chat, requests, payments, purchase_orders, receipts,
           invoices, reports, audit_log, compliance, docs (read-only)

* Internal-only: demo, solana_ct_demo, debug (NEXT_PUBLIC_INTERNAL_MODE=1)


3.4 SEPARATION OF DUTIES (SoD)
--------------------------------------------------------------------------------
- Requester CANNOT approve their own request
- One decision per actor per request (unique requestId + actorUserId)
- Pay: ADMIN only
- Vendor wallet/status: ADMIN only
- Approval policy, spend policy: ADMIN only
- Reconcile: ADMIN only
- Treasury payouts: Require approval (configurable threshold)
- Payout approval: Different user than creator

5.1 TOKEN-2022 (SPL Token Extensions)
--------------------------------------------------------------------------------
Program ID: TokenzQdBNbLqP5VEhdkAS6EPFLC1PHnBqCXEpPxuEb

Features used:
- Standard Token-2022 mint creation
- Required Memo Transfer extension - every transfer MUST include memo
- Confidential Transfer (CT): Intended but DISABLED - Solana ZK ElGamal audit
  in progress. Create mint, init accounts, mint-to-treasury work. Deposit,
  apply-pending, ct-transfer, withdraw return 503 + "CT disabled" message.

Decimals: 9

- Optional HTTP Basic auth (SOLANA_RPC_BASIC_USER / SOLANA_RPC_BASIC_PASS)

- 10-second timeout (withTimeout helper)
- RpcNotConfiguredError for graceful degradation
- redactRpcUrlForLog() - never logs full URL or credentials

Memo placed immediately before transfer to satisfy Required Memo extension.


5.5 EXPENSE PAYMENT FLOW (Token-2022 Transfer)
--------------------------------------------------------------------------------
1. Request must be APPROVED with sufficient approvals
2. Receipt required if policy mandates
3. Budget check; over-budget needs override note if allowAdminOverrideOverBudget
4. Vendor must be ACTIVE with ownerPubkey set
5. Token-2022 transfer via transferWithMemo (SPL Memo + Transfer)
6. Memo format: "KharchaPay Request {requestId} [{orgSlug}]"


5.6 ORG SETUP PAYMENT (Solana)
--------------------------------------------------------------------------------
Flow:
1. Create org (status PENDING_PAYMENT)
2. OrgSetupPaymentIntent: requiredLamports, depositPubkey, treasuryPubkey
3. User sends SOL to depositPubkey
4. submit-tx verifies on-chain, marks intent PAID
5. Org status → PENDING_TERMS → ACTIVE after terms acceptance

Config:
ORG_CREATE_FEE_SOL (default 0.006)
ORG_CREATE_TREASURY_PUBKEY
Bypass available for dev/demo when no mainnet payment
Rate limits: MAX_CHECK = 30, MAX_SUBMIT_TX = 10


5.7 RECONCILIATION (On-Chain Verification)
--------------------------------------------------------------------------------
File: src/lib/solana/verify-payment.ts (350 lines)

verifyPaymentOnChain(requestId):
1. Tx exists and confirmed
2. Memo matches "KharchaPay Request {requestId}"
3. Token program + mint = Token-2022
4. Source = org treasury, dest = vendor token account
5. Amount = request amountMinor

Handles both legacy and versioned transactions (accountKeys/staticAccountKeys)
Supports dependency injection of mock RPC for testing

Status: VERIFIED | WARNING | FAILED | PENDING
Error codes: RPC_TIMEOUT, RPC_UNAVAILABLE, RPC_ERROR

7. CORE FEATURE FLOWS
================================================================================

7.1 EXPENSE REQUEST LIFECYCLE
--------------------------------------------------------------------------------
DRAFT → (Submit) → PENDING → (Approve x N) → APPROVED → (Pay) → PAID
                   \_ (Reject) → REJECTED

- Submit: Snapshots requiredApprovals from ApprovalPolicy tiers
- Approve: One decision per actor; SoD (requester cannot approve)
- Pay: ADMIN only; requires receipt (if policy), budget check, vendor ACTIVE


7.2 PROCURE-TO-PAY LIFECYCLE
--------------------------------------------------------------------------------
PO: DRAFT → ISSUED → PARTIALLY_RECEIVED → RECEIVED → CLOSED
GRN: DRAFT → SUBMITTED → ACCEPTED
Invoice: DRAFT → SUBMITTED → NEEDS_VERIFICATION → VERIFIED → APPROVED/REJECTED

2-way match: PO ↔ Invoice
3-way match: PO ↔ Invoice ↔ GRN

OrgMatchTolerance: qtyTolerancePct, priceTolerancePct, amountTolerancePct
Bulk operations: bulk-verify, bulk-assign


7.3 VENDOR LIFECYCLE
--------------------------------------------------------------------------------
Status: DRAFT → ONBOARDING → ACTIVE | BLOCKED | ARCHIVED | INACTIVE

- Only ACTIVE vendors: new requests, payment
- APPROVER can set BLOCKED
- VendorBankChangeRequest: dual approval (OrgVendorPolicy)
- VendorOnboardingCase: checklist, risk level
- VendorFiatPayoutProfile: Circle beneficiary for fiat off-ramping
- VendorPayoutProfile: POST .../vendors/[id]/payout-profile


7.4 NOTIFICATIONS
--------------------------------------------------------------------------------
Types: REQUEST_NEEDS_APPROVAL, REQUEST_APPROVED, REQUEST_REJECTED, REQUEST_PAID
Triggers: Submit → notify approvers; Approve/Reject → notify requester; Pay → notify requester
API: GET /api/notifications, POST /api/notifications/[id]/read
Treasury: Treasury-specific notifications via treasury-notifier.ts

9. SECURITY DETAILS
================================================================================

9.1 SECURITY HEADERS (middleware.ts)
--------------------------------------------------------------------------------
Applied to all routes except static assets:
- Content-Security-Policy: default-src 'self', frame-ancestors 'none',
  base-uri 'self', form-action 'self', connect-src 'self'
- X-Content-Type-Options: nosniff
- Referrer-Policy: strict-origin-when-cross-origin
- Permissions-Policy: camera=(), microphone=(), geolocation=()
- X-Frame-Options: DENY
- Production: Strict-Transport-Security: max-age=31536000; includeSubDomains; preload


9.2 RECEIPT UPLOAD
--------------------------------------------------------------------------------
Allowlist: .pdf, .jpg, .jpeg, .png, .webp
Magic-byte validation (no trust of Content-Type/filename):
  - PDF: %PDF-, JPEG: 0xFF 0xD8, PNG: 8-byte header, WebP: RIFF...WEBP
Max size: 5MB
Storage: Random filename, RECEIPT_STORAGE_DIR (outside public/)
Rate limit: 10 uploads/min (receipt:upload)
Download: GET /api/receipts/[id] - auth + org read (AUDITOR can view)
Startup check: receipt-storage-init.ts fails fast if storage dir not writable


9.3 RATE LIMITING
--------------------------------------------------------------------------------
Two layers:

Basic (rate-limit.ts): In-memory Map, 60-second window
  - MAX_LOGIN_ATTEMPTS = 10 per email
  - MAX_CHECK = 30 (org setup check)
  - MAX_SUBMIT_TX = 10 (org setup submit)

Advanced (rate-limiter.ts): Per-route-group with IP identification
  | Route Group           | Limit   |
  |-----------------------|---------|
  | auth:login            | 10/min  |
  | auth:forgot-password  | 5/min   |
  | auth:reset-password   | 5/min   |
  | demo:start            | 10/min  |
  | demo:reset            | 3/min   |
  | receipt:upload        | 10/min  |
  | receipt:download      | 60/min  |
  | reconcile:run         | 5/min   |
  | export                | 20/min  |
  | global                | 200/min |

IP extraction: CF-Connecting-IP > X-Forwarded-For > X-Real-IP
Returns: { limited, retryAfterSeconds }

Demo: 60-second cooldown per user for demo resets (demo-rate-limit.ts)


9.4 ENCRYPTION AT REST
--------------------------------------------------------------------------------
File: src/lib/encryption.ts
Algorithm: AES-256-GCM, 12-byte IV, 16-byte auth tag
Key: ENCRYPTION_KEY (64-char hex) in production; scryptSync(JWT_SECRET) fallback in dev
Used for: QuickBooks OAuth tokens, treasury keypairs


9.5 AUDIT LOG
--------------------------------------------------------------------------------
Append-only AuditEvent. Actions: ORG_CREATED, MEMBER_ADDED, DEPT_CREATED,
BUDGET_UPSERTED, REQUEST_CREATED/UPDATED/SUBMITTED/APPROVED/REJECTED/PAID,
RECEIPT_UPLOADED, VENDOR_*, SPEND_POLICY_UPDATED, PAYMENT_BLOCKED,
AUDIT_RETENTION_RUN, RECONCILIATION_*, etc.

Retention: OrgAuditRetention.retentionDays (30-3650); ADMIN runs cleanup.

Treasury audit: Separate TreasuryAuditLog with Prisma middleware preventing
mutation of existing records (audit-immutability.ts).


9.6 AUDIT IMMUTABILITY
--------------------------------------------------------------------------------
Prisma middleware (audit-immutability.ts) prevents UPDATE and DELETE on
TreasuryAuditLog and TreasuryLedgerEntry models. Append-only by design.

18. KEY ARCHITECTURAL PATTERNS
================================================================================

1. Monolith-first: apps/web serves 54 pages + 184 API route handlers.
   apps/api is optional (health endpoint only).

2. Organization as hub: Nearly every model connects back to Organization
   via orgId with onDelete: Cascade.

3. Transactional outbox: OutboxEvent → WebhookDeliveryAttempt → WebhookEndpoint
   for reliable webhook delivery with retry.

4. Event-driven treasury: 24 event types via TreasuryEvent with dedup keys,
   SSE streaming to UI, and webhook forwarding.

5. Double-entry ledger: TreasuryLedgerEntry with 6 accounts and immutability
   middleware preventing mutations.

6. Named Prisma relations: User has named relations (Requester, ApprovalActor,
   POCreatedBy, GRNReceivedBy, InvoiceCreatedBy/Verified/Assigned) to avoid
   ambiguity.

7. Vendor cascade: Vendor → Contact[], Document[], PaymentMethod[],
   BankChangeRequest[], OnboardingCase[], FiatPayoutProfile.

8. onDelete strategies: Cascade (most), Restrict (Vendor on Request/PO/Invoice),
   SetNull (Department on PO/Invoice).

9. Provider abstraction: Payout providers (Circle) behind interface with
   capability queries and rail validation.

10. Safety-first treasury: Circuit breakers, global/per-rail/per-provider
    pause toggles, spend policies, approval thresholds, reconciliation checks.

11. Interactive tours: Role-based guided tours (ADMIN, APPROVER, STAFF) with
    step-by-step walkthroughs, localStorage persistence, and reset via user menu.
